# Vortex Live

سیستم کنفرانس ویدیویی زنده — Express + Socket.IO + WebRTC، بدون دیتابیس (ذخیره‌سازی روی فایل JSON).

## نصب و اجرا

```bash
npm install
npm start
```

سرور روی پورت `process.env.PORT` (پیش‌فرض 3000) بالا می‌آید.

## ⚠️ اگر روی cPanel، اپ رو زیر یک زیرمسیر گذاشتید (مثل `/vortex` یا `/live-streem`)

توی صفحهٔ **Setup Node.js App**، فیلد **«Application URL»** رو هرچی گذاشته باشید (مثلاً `vortex`)، این دقیقاً همون مسیریه که Passenger بدون حذف‌کردنش به اپ پاس می‌ده — یعنی اگه چیزی برای این مسیر تعریف نشده باشه، با `Cannot GET /vortex/` مواجه می‌شید.

برای رفعش، یک متغیر محیطی به اپ اضافه کنید (همون صفحهٔ Setup Node.js App، بخش **Environment Variables**):

```
BASE_PATH=vortex
```

(دقیقاً همون مقداری که تو «Application URL» گذاشتید، با یا بدون `/` در ابتدا فرقی نداره). بعد از اضافه‌کردنش، دکمهٔ **Restart** رو بزنید. از این به بعد همهٔ فایل‌های استاتیک، API و اتصال سوکت به‌درستی زیر همون مسیر سرو می‌شن — چه اپ رو زیر یک زیرپوشه گذاشته باشید چه روی ریشهٔ دامنه (که در اون صورت این متغیر رو اصلاً لازم ندارید).

## ورود به پنل مدیریت

آدرس: `https://دامنه-شما/admin`
یوزرنیم/پسورد پیش‌فرض: **admin / admin** — حتماً از همان صفحه، بخش «تغییر یوزرنیم و پسورد مدیر» را عوض کنید.

از همین‌جا اتاق جدید می‌سازید، لینک آن را کپی می‌کنید، و هر وقت خواستید آن را **می‌بندید** — بعد از بستن، آن لینک برای همیشه غیرفعال می‌شود و کسی نمی‌تواند با آن وارد شود. **هیچ لابی پیش‌فرضی وجود ندارد**؛ باز کردن آدرس اصلی سایت بدون کد اتاق، صفحه‌ی «اتاقی برای ورود وجود ندارد» را نشان می‌دهد.

نکتهٔ مهم: وقتی مدیر، لینک یک اتاق را در همان مرورگرِ لاگین‌شده باز می‌کند، به‌طور خودکار با اختیارات مدیر وارد می‌شود (نیازی به لاگین دوباره داخل اتاق نیست) و پنل «کاربران» با دکمه‌های قطع صدا / قطع دوربین / اخراج برایش فعال می‌شود.

## دربارهٔ تشخیص ضبط صفحه (نکته صادقانه)

خواسته بود که اگر کسی با گوشی (iOS یا Android) از جلسه فیلم بگیرد، مدیر متوجه شود. **این از طریق یک وب‌سایت فنی ممکن نیست** — نه مرورگر و نه هیچ API استانداردی، به سایت اجازه نمی‌دهد بفهمد کاربر از ضبط‌کنندهٔ داخلی گوشی (مثل Control Center در iOS) استفاده می‌کند؛ این یک محدودیت امنیتی و حریم‌خصوصی سیستم‌عامل است، نه محدودیت این پروژه. ادعای «تشخیص ضبط» در چنین ابزارهایی معمولاً واقعی نیست.

به‌جای آن، دو تدبیر واقعی پیاده شده:
1. **واترمارک زنده روی ویدیوی هرکس** (نام کاربر + تاریخ) که اگر جایی درز کند، قابل ردیابی باشد.
2. **قابلیت «ضبط جلسه» فقط برای مدیر** فعال است (دکمهٔ «شروع ضبط» فقط در پنل مدیر دیده می‌شود) و از طریق `getDisplayMedia` + `MediaRecorder` خروجی `.webm` روی همان دستگاه مدیر ذخیره می‌شود. کاربران عادی اصلاً به این دکمه دسترسی ندارند.

## اتصال به OBS

هر اتاق یک لینک اختصاصی OBS دارد (در پنل مدیریت، دکمهٔ «لینک OBS»)، به شکل:
```
https://دامنه-شما/obs?room=CODE
```
این صفحه بدون هیچ دکمه/چت/UI است و فقط تصویر شرکت‌کننده‌ها را نمایش می‌دهد — آن را به‌عنوان **Browser Source** در OBS اضافه کنید (Sources → + → Browser، آدرس بالا را بگذارید، عرض/ارتفاع دلخواه). این ورودی، صدا و تصویر هیچ‌کس را ارسال نمی‌کند (فقط دریافت‌کننده است) پس در لیست کاربران دیگران دیده نمی‌شود.

برای پس‌زمینهٔ شفاف/کروماکی، `?transparent=1` را به انتهای لینک اضافه کنید (پس‌زمینهٔ صفحه مشکی خالص می‌شود تا با Chroma Key در OBS جفت شود؛ توجه: این آلفای واقعی نیست، چون یک تگ `<video>` در مرورگر کانال شفافیت ندارد — کروماکی روی مشکی، راه‌حل عملی است).

> اگر منظورتان از «webart» چیز مشخص دیگری بود (مثلاً پروتکل WHIP یا خروجی RTMP مستقیم به یوتیوب/اینستاگرام)، آن یک قابلیت جداگانه است که به یک سرور رسانه‌ای (SFU/ffmpeg relay) نیاز دارد و در این نسخه ساخته نشده — بگویید دقیقاً چه می‌خواهید تا برایش برنامه‌ریزی کنم.

## نکتهٔ مهم دربارهٔ هاست cPanel (چرا اتاق‌ها قبلاً وصل نمی‌شدند)

اکثر هاست‌های cPanel، اپ Node.js را از طریق **Phusion Passenger** اجرا می‌کنند که معمولاً **ارتقاء اتصال به WebSocket را به‌درستی پراکسی نمی‌کند**. نتیجه این می‌شود که کاربر وارد اتاق می‌شود (چون سیگنالینگ روی HTTP polling کار می‌کند) ولی تصویر/صدا (که برای اتصال مستقیم WebRTC نیاز به هماهنگی پایدار دارد) وصل نمی‌شود یا خیلی ناپایدار است.

در این نسخه:
- کلاینت به‌صورت صریح با `transports: ['polling','websocket']` تنظیم شده تا اگر websocket پراکسی نشود، حداقل polling کار کند.
- منطق اتصال مجدد (reconnect) و «خوددرمانی» peer connection اضافه شده که با هر آپدیت لیست کاربران، اتصالات ازدست‌رفته را دوباره برقرار می‌کند.
- سرور STUN عمومی گوگل + امکان تنظیم **TURN** از طریق متغیرهای محیطی زیر اضافه شده (برای تماس بین کاربرانی که پشت NAT/فایروال سخت‌گیرانه‌اند، TURN تقریباً ضروری است):
  ```
  TURN_URL=turn:your-turn-server:3478
  TURN_USERNAME=xxxx
  TURN_CREDENTIAL=xxxx
  ```
  بدون TURN، در بسیاری شبکه‌ها (خصوصاً موبایل/CGNAT) دو کاربر «وارد اتاق می‌شوند ولی همدیگر را نمی‌بینند» — همان مشکلی که گزارش شده بود. توصیه می‌شود یک TURN شخصی (مثلاً coturn روی یک VPS ارزان) راه‌اندازی کنید.
- اگر هاست شما امکان فعال‌سازی WebSocket پراکسی را دارد (برخی پلن‌های cPanel با `.htaccess` یا تنظیم Apache `mod_proxy_wstunnel` این کار را ممکن می‌کنند)، حتماً فعالش کنید؛ پایداری بسیار بهتر می‌شود.

## ساختار پروژه

```
server.js              بک‌اند Express + Socket.IO + REST API
public/index.html       صفحهٔ ورود/کنفرانس
public/js/app.js         منطق کلاینت اتاق (WebRTC mesh، چت، مدیریت)
public/obs/index.html    نمای مخصوص OBS
public/js/obs.js
public/admin/index.html  پنل مدیریت
public/js/admin.js
public/css/style.css     سیستم طراحی (تم اتاق کنترل پخش زنده)
public/assets/poster.svg برند/پوستر (SVG برداری + خروجی PNG)
data/                    rooms.json و settings.json — به‌صورت خودکار ساخته می‌شود
```

## آنچه در این نسخه رفع/اضافه شد

1. طراحی کاملاً واکنش‌گرا (گرید ویدیو با `auto-fit`، هدر/فوتر/سایدبار همه موبایل-محور، سایدبار روی موبایل به‌صورت Bottom Sheet).
2. بازطراحی کامل UI با یک سیستم طراحی مشخص (نه Tailwind پیش‌فرض CDN).
3. رفع اتصال بین کاربران با لینک‌های مختلف (TURN + polling fallback + خوددرمانی اتصال، توضیح داده‌شده بالا).
4. حذف کامل `<form>` از چت و ورود — دیگر هیچ چیزی نمی‌تواند صفحه را رفرش کند؛ پیام‌ها آنی روی سوکت ارسال می‌شوند.
5. نام‌نویسی اجباری قبل از ورود (سمت کلاینت و سرور).
6. پنل مدیریت با لیست کاربران زنده + قطع صدا/دوربین هرکس + اخراج.
7. اتاق‌ها فقط توسط مدیر ساخته می‌شوند؛ با «بستن» اتاق، لینک آن برای همیشه غیرفعال می‌شود (بدون لابی پیش‌فرض).
8. ضبط فقط برای مدیر + واترمارک شناسایی (به‌جای «تشخیص ضبط» که فنی ممکن نیست — توضیح در بالا).
9. لاگین مدیر با یوزر/پسورد پیش‌فرض admin/admin، قابل تغییر از پنل.
10. اتصال به OBS از طریق لینک Browser Source اختصاصی هر اتاق.
